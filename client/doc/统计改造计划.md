# 统计改造计划

## 背景

Analytics 看板需要同时支持近期分析和长期累计统计。Cloudflare Analytics Engine 适合保存详细埋点并做近期查询，但数据保留期有限，不能单独承担历史总数。

旧 Queue 方案已经废弃：`/track` 每条事件投递 Cloudflare Queue 会直接消耗 Queue 用量，成本和免费层风险不可控。当前最终方案不使用 Queue。

## 最终目标

1. `/track` 只写 Analytics Engine，不在上报热路径写 D1 或 Queue。
2. Worker Cron 每天按 `Asia/Shanghai` 业务日汇总昨日 Analytics Engine 聚合结果到 D1。
3. D1 不保存原始事件明细，不保存明文 `client_id`。
4. D1 只保存每日聚合结果和匿名 SHA-256 hash 去重索引。
5. Dashboard 历史总数读 D1，最近 7/30/90 天仍读 Analytics Engine。
6. 旧线上 Queue 只能在去 Queue Worker 部署成功后再删除。

## 总体架构

```text
客户端埋点
  -> POST /track
    -> Analytics Engine：保存详细事件，用于最近事件、近期分析和每日汇总来源

Worker Cron：每天 UTC 18:15 / 北京时间 02:15
  -> 查询昨日 Analytics Engine 聚合结果
  -> ANALYTICS_DB D1：每日汇总表 + 匿名客户端 hash 索引

Dashboard
  -> 历史总数：读取 D1
  -> 最近 7/30/90 天：读取 Analytics Engine
```

## 数据口径

| 类型 | 数据源 | 说明 |
| --- | --- | --- |
| 详细埋点 | Analytics Engine | `/track` 唯一写入目标，保留近期可查询明细 |
| 历史累计 | D1 | 每日汇总表按 `source = 'rollup'` 求和 |
| 长期去重客户端 | D1 | `sha256(projectName + ':' + client_id)`，不保存明文 ID |
| 维度去重客户端 | D1 | 页面、版本、配置、模型、资源等维度使用匿名 hash 索引 |
| 最近 7/30/90 天 | Analytics Engine | 保持灵活查询，不依赖 D1 |

所有业务日期统一使用 `Asia/Shanghai`，格式 `YYYY-MM-DD`。

## Worker 写入流程

### `/track`

1. 解析请求 JSON。
2. 规范化字段和 token 用量。
3. 校验 `projectName`、`event`、必要字段。
4. 调用 `env.ANALYTICS.writeDataPoint()` 写 Analytics Engine。
5. 返回 `{ "code": 0 }`。

`/track` 不生成 `event_id`，不投递 Queue，不写 D1。客户端仍保持 fire-and-forget，埋点失败不能影响业务。

### Cron 汇总

`wrangler.jsonc` 配置：

```jsonc
"triggers": {
  "crons": [
    "15 18 * * *"
  ]
}
```

Cron 任务执行 `rollupYesterdayForAllProjects()`：

1. 按上海业务日计算昨日日期。
2. 从 Analytics Engine 查询昨日出现过的项目名。
3. 对每个项目执行 `rollupAnalyticsDay()`。
4. 删除该项目该日期旧的 D1 daily 汇总行。
5. 写入每日汇总、页面、版本、配置、模型、资源统计。
6. 写入匿名客户端索引和匿名维度客户端索引。

## D1 表设计

当前新增迁移文件：`analytics/worker/analytics-migrations/0002_create_analytics_daily_rollups.sql`。

| 表 | 用途 |
| --- | --- |
| `analytics_daily_summary` | 每项目每天的事件量、活跃客户端、token、首次/最近上报时间 |
| `analytics_daily_event_client_stats` | 每项目每天每类事件的去重客户端数；`app_open` 用作 DAU/打开客户端口径 |
| `analytics_daily_page_stats` | 每日页面访问量和当日触达客户端数 |
| `analytics_daily_version_stats` | 每日版本分布和各事件类型计数 |
| `analytics_daily_config_stats` | 每日配置使用上报次数和当日客户端数 |
| `analytics_daily_model_stats` | 每日模型请求、token 用量和当日客户端数 |
| `analytics_daily_resource_stats` | 每日资源点击量和当日客户端数 |
| `analytics_client_index` | 匿名客户端生命周期索引 |
| `analytics_dimension_client_index` | 匿名维度客户端去重索引 |
| `analytics_dimension_values` | 维度 key 到展示 label 的映射，来自旧 migration 但新方案继续使用 |
| `analytics_meta` | 汇总元信息，来自旧 migration 但新方案继续使用 |

旧 Queue 版表可以保留，不作为新查询权威来源：

```text
analytics_clients
analytics_daily_client_activity
analytics_monthly_*
analytics_dimension_clients
analytics_dimension_client_totals
analytics_processed_events
```

## Dashboard 口径

| 页面 | 最近 7/30/90 天 | 历史总数 |
| --- | --- | --- |
| 概览 | 范围分析仍可读 Analytics Engine | 长期累计读 D1 |
| 访问分析 | Analytics Engine | D1 daily 表 + 匿名维度索引 |
| 配置使用 | Analytics Engine | D1 daily 表 + 匿名维度索引 |
| 模型使用 | Analytics Engine | D1 daily 表 + 匿名维度索引 |
| 资源管理 | Analytics Engine | D1 daily 表 |

注意：Cron 默认只汇总到昨日。今日活跃等需要当天数据的指标，如果只读 D1，会在当天汇总前存在延迟；近期分析应继续使用 Analytics Engine。

## 回填

回填脚本：`analytics/scripts/backfill-analytics-rollups.mjs`。

示例：

```powershell
cd analytics\worker
$env:ACCOUNT_ID = "<Cloudflare Account ID>"
$env:ANALYTICS_API_TOKEN = "<Analytics Engine Read Token>"
npm run backfill:analytics -- --project yibiao-client --start 2026-03-15 --end 2026-06-12 --remote
```

dry-run：

```powershell
cd analytics\worker
npm run backfill:analytics -- --project yibiao-client --start 2026-03-15 --end 2026-06-12 --dry-run
```

参数：

| 参数 | 说明 |
| --- | --- |
| `--project` | 项目名，默认 `yibiao-client` |
| `--start` | 回填起始业务日期 |
| `--end` | 回填结束业务日期，默认昨日 |
| `--remote` | 写远程 D1 |
| `--local` | 写本地 Wrangler D1 |
| `--dry-run` | 只打印覆盖日期，不查询 Analytics Engine，不写 D1 |
| `--allow-current-day` | 允许回填今天，仅在确认当天数据稳定时使用 |

回填写入口径统一为 `source = 'rollup'`，和 Cron 汇总一致。

## 部署顺序

1. 设置 Cloudflare 凭据：`CLOUDFLARE_API_TOKEN`、`CLOUDFLARE_ACCOUNT_ID`。
2. 在 `analytics/worker` 下运行 `npm run setup:analytics-storage`，创建或复用 `ANALYTICS_DB` 并执行 migration。
3. 部署当前去 Queue Worker。
4. 验证 `/track` 返回 `{ "code": 0 }`。
5. 如线上曾部署 Queue 方案，再删除 `openbidkit-analytics-rollup` Queue 和相关 binding/consumer。
6. 按需要运行历史回填脚本。

不要在去 Queue Worker 部署前删除线上 Queue binding，否则旧线上 `/track` 会因为缺少 `ANALYTICS_ROLLUP_QUEUE` 失败。
